### **[Claude Code 작동 방식](https://code.claude.com/docs/ko/how-claude-code-works)**

- Claude에게 작업을 주면 세 가지 단계를 거칩니다: **컨텍스트 수집**, **작업 수행**, **결과 검증**.
  - 결과 검증 제대로 안하던데 나름대로 하는거였군..
- **기본 기능 확장:** 내장 도구는 기초입니다. **[skills](https://code.claude.com/docs/ko/skills)**로 Claude가 알 수 있는 것을 확장하고, **[MCP](https://code.claude.com/docs/ko/mcp)**로 외부 서비스에 연결하고, **[hooks](https://code.claude.com/docs/ko/hooks)**로 워크플로우를 자동화하고, **[subagents](https://code.claude.com/docs/ko/sub-agents)**로 작업을 오프로드할 수 있습니다. 이러한 확장은 핵심 에이전트 루프 위에 계층을 형성합니다. 필요에 맞는 확장을 선택하는 방법은 **[Claude Code 확장](https://code.claude.com/docs/ko/features-overview)**을 참조하십시오.
  - hooks는 처음들었네.
- 원본 세션에 영향을 주지 않고 다른 접근 방식을 시도하려면 `--fork-session`플래그를 사용합니다. `claude --continue --fork-session`
  - 복잡한 테스트를 할 때 유용할듯? A방식 시도해보고 안되면 A되돌리고 B방식 시도하고 B 되돌리고 이러면 세션이 엄청 길어지고 알아보기도 힘듬.
- 압축 중에 보존되는 것을 제어하려면 CLAUDE.md에 “Compact Instructions” 섹션을 추가하거나 **`/compact`**를 포커스와 함께 실행합니다 (예: `**/compact focus on the API changes`\*\*).
  - 그냥 compact를 하면 전체를 요약하는데 뒤에 문자열을 넣으면 그 부분은 많이 살리고 나머지는 많이 요약함.
- **`/context`**를 실행하여 공간을 사용하는 것을 확인합니다.
  - 이거 실행해서 skill에 어느정도 token이 들어가는 것도 확인할 수 있음.
- MCP servers는 모든 요청에 도구 정의를 추가하므로 몇 개의 서버가 작업을 시작하기 전에 상당한 context를 소비할 수 있습니다. **`/mcp`**를 실행하여 서버별 비용을 확인합니다.
- 수동으로 호출하는 skills의 경우 `disable-model-invocation: true`
  - 컨텍스트 아끼기 위해서 중요할듯. skill을 한달에 한번씩 검토해주는 skill을 만들고 여기서 이걸 쓰면 좋으려나..
- **[Subagents](https://code.claude.com/docs/ko/sub-agents)**는 주 대화와 완전히 분리된 자신의 새로운 context를 얻습니다. 그들의 작업은 context를 부풀리지 않습니다. 완료되면 요약을 반환합니다. 이 격리가 subagents가 긴 세션에 도움이 되는 이유입니다.
  - 인정..
- **`/init`**은 프로젝트를 위한 CLAUDE.md 생성을 안내합니다
- **`/agents`**는 사용자 정의 subagents 구성을 도와줍니다
- **`/doctor`**는 설치의 일반적인 문제를 진단합니다
- 초기 프롬프트가 정확할수록 필요한 수정이 적습니다. 특정 파일을 참조하고, 제약 조건을 언급하고, 예제 패턴을 지적합니다.
- Claude는 자신의 작업을 확인할 수 있을 때 더 잘 수행합니다. 테스트 케이스를 포함하고, 예상 UI의 스크린샷을 붙여넣거나, 원하는 출력을 정의합니다.
  - openapi claude code 자동화가 이거에 해당하는 좋은 예시일듯.
  - UI는 chrome mcp붙여서 테스트하라고 하면 좋음. 만약 이전과 달라져야하지 않는다면 dev서버와 로컬서버 띄워두고 두개비교하라고 하는 것 도 괜찮음.
- 복잡한 문제의 경우 연구와 코딩을 분리합니다.
  - plan 모드는 너무 당연한거고 plan 모드로 실행하지 않아도 될정도의 복잡도인경우에도 분리하면 괜찮을듯?

### [지침 및 메모리 저장](https://code.claude.com/docs/ko/memory)

- 빌드 및 테스트 명령, 코딩 표준, 아키텍처 결정, 명명 규칙 및 일반적인 워크플로우. 이러한 지침은 버전 제어를 통해 팀과 공유되므로 개인 선호도보다는 프로젝트 수준의 표준에 중점을 두세요.
  - 요즘 링크드인글보면 이런거 없는게 더 좋다는데.. 클로드 공식문서는 아닌데.. 뭐가 맞는걸까? 내생각엔 빌드 및 테스트는 유의미하다고 생각함. 왜냐면 package.json을 결국 읽어야하니깐. 근데 만약 claude code가 기본적으로 package.json을 읽는다면 claude.md에 지침을 넣을 필요가 없겠지. 근데 내 생각엔 package.json을 기본적으로 읽게 할 것 같기는 않아. 이건 잘못 만들어진거 아닌가… 실제로 테스트도 좀 해봐야겠다. > 자동으로 읽지는 않는다고함. 그러면 결국 claude.md가 길어지는게 문제지 짧다면 문제가 없을듯?
- **크기**: CLAUDE.md 파일당 200줄 이하를 목표로 합니다. 더 긴 파일은 더 많은 컨텍스트를 소비하고 준수를 줄입니다. 지침이 커지면 **[가져오기](https://code.claude.com/docs/ko/memory#import-additional-files)**를 사용하거나 `**[.claude/rules/](https://code.claude.com/docs/ko/memory#organize-rules-with-clauderules)`\*\* 파일로 분할합니다.
- CLAUDE.md 파일은 `**@path/to/import**` 구문을 사용하여 추가 파일을 가져올 수 있습니다. 가져온 파일은 확장되어 이를 참조하는 CLAUDE.md와 함께 시작 시 컨텍스트에 로드됩니다.

  ```jsx
  프로젝트 개요는 @README를 참조하고 이 프로젝트의 사용 가능한 npm 명령은 @package.json을 참조합니다.

  # 추가 지침
  - git 워크플로우 @docs/git-instructions.md
  ```

  - 이거 그냥 경로를 넣으면 필요할때만 파일을 load하는거고 항상 넣을 때 @를 사용하는거야.

- 규칙(rules)은 모든 세션 또는 일치하는 파일이 열릴 때 컨텍스트에 로드됩니다. 항상 컨텍스트에 있을 필요가 없는 작업별 지침의 경우 대신 **[skills](https://code.claude.com/docs/ko/skills)**를 사용하세요. 이는 호출할 때 또는 Claude가 프롬프트와 관련이 있다고 판단할 때만 로드됩니다.
  - 뭐가 다른거지? skills도 기본적으로 일반 컨텍스트에서 description같은거보고 읽어서 매핑되면 실행하잖아. ⇒ rule은 트리거가 파일 패턴이야. 이게 달라.

  ```jsx
  ---
  paths:
    - "src/api/**/*.ts"
  ---

  # API 개발 규칙

  - 모든 API 엔드포인트는 입력 검증을 포함해야 합니다
  - 표준 오류 응답 형식을 사용합니다
  - OpenAPI 문서 주석을 포함합니다
  ```

- `**.claude/rules/**` 디렉토리는 심볼릭 링크를 지원하므로 공유 규칙 세트를 유지하고 여러 프로젝트에 링크할 수 있습니다. 심볼릭 링크가 해석되고 정상적으로 로드되며 순환 심볼릭 링크가 감지되고 정상적으로 처리됩니다.
  ```jsx
  ln -s ~/shared-claude-rules .claude/rules/shared
  ln -s ~/company-standards/security.md .claude/rules/security.md
  ```
- 대규모 모노레포에서 상위 CLAUDE.md 파일에는 작업과 관련이 없는 지침이 포함될 수 있습니다. `**claudeMdExcludes**` 설정을 통해 경로 또는 glob 패턴으로 특정 파일을 건너뛸 수 있습니다.
   제외가 머신에 로컬로 유지되도록 `**.claude/settings.local.json`**에 추가합니다:
  `jsx { "claudeMdExcludes": [ "**/monorepo/CLAUDE.md", "/home/user/monorepo/other-team/.claude/rules/\*\*" ] }`
  ```
  - 이건 진짜 대규모팀에서 다른 팀의 코드를 건드리지 않아야할 때나 사용할만함. 프론트 백엔드가 같은 모노레포인 경우도 그렇고.
  ```
- 자동 메모리를 통해 Claude는 아무것도 작성하지 않고도 세션 간에 지식을 축적할 수 있습니다.
  ```jsx
  ~/.claude/projects/<project>/memory/
  ├── MEMORY.md          # 간결한 인덱스, 모든 세션에 로드됨
  ├── debugging.md       # 디버깅 패턴에 대한 상세 노트
  ├── api-conventions.md # API 설계 결정
  └── ...                # Claude가 만드는 다른 주제 파일
  ```

  - claude code 많이 사용한 프로젝트에 들어가봤는데 생각보다 별거 없었음..
- **`MEMORY.md`**의 처음 200줄이 모든 대화의 시작 시 로드됩니다. 200줄을 초과하는 콘텐츠는 세션 시작 시 로드되지 않습니다. Claude는 상세한 노트를 별도의 주제 파일로 이동하여 **`MEMORY.md`**를 간결하게 유지합니다.
- `**/memory`\*\* 명령은 현재 세션에 로드된 모든 CLAUDE.md 및 규칙 파일을 나열하고, 자동 메모리를 켜거나 끌 수 있으며, 자동 메모리 폴더를 열 수 있는 링크를 제공합니다.
  - user memory, project memory 두 계층으로 존재함.
  - claude slack 알림 플러그인 사용중인데 이게 user memory에 들어가있었다.

### **[Claude Code 모범 사례](https://code.claude.com/docs/ko/best-practices)**

- Claude가 규칙에도 불구하고 계속 원하지 않는 작업을 수행하면 파일이 너무 길어서 규칙이 손실되고 있을 가능성이 있습니다. Claude가 CLAUDE.md에서 답변된 질문을 하면 표현이 모호할 수 있습니다. CLAUDE.md를 코드처럼 취급하십시오: 문제가 발생하면 검토하고, 정기적으로 정리하고, 변경 사항을 관찰하여 Claude의 동작이 실제로 변경되는지 테스트하십시오.
- 강조(예: “IMPORTANT” 또는 “YOU MUST”)를 추가하여 지시사항을 조정하면 준수를 개선할 수 있습니다.

| **✅ 포함**                             | **❌ 제외**                              |
| --------------------------------------- | ---------------------------------------- |
| Claude가 추측할 수 없는 Bash 명령       | Claude가 코드를 읽어서 파악할 수 있는 것 |
| 기본값과 다른 코드 스타일 규칙          | Claude가 이미 알고 있는 표준 언어 규칙   |
| 테스트 지시사항 및 선호하는 테스트 러너 | 상세한 API 문서(대신 문서 링크)          |
| 저장소 에티켓(분기 이름 지정, PR 규칙)  | 자주 변경되는 정보                       |
| 프로젝트에 특정한 아키텍처 결정         | 긴 설명 또는 튜토리얼                    |
| 개발자 환경 특이성(필수 환경 변수)      | 자명한 관행(예: “깨끗한 코드 작성”)      |
| 일반적인 함정 또는 명백하지 않은 동작   | 파일별 코드베이스 설명                   |

- **[Hooks](https://code.claude.com/docs/ko/hooks-guide)**는 Claude의 워크플로우의 특정 지점에서 자동으로 스크립트를 실행합니다. 권고적인 CLAUDE.md 지시사항과 달리 hooks는 결정론적이며 작업이 발생함을 보장합니다.
- `**Esc**`: `**Esc**` 키로 Claude의 작업을 중간에 중지하십시오. Context는 보존되므로 방향을 바꿀 수 있습니다.
- `**Esc + Esc` 또는 `/rewind**`: `**Esc`**를 두 번 누르거나 **`/rewind`\*\*를 실행하여 rewind 메뉴를 열고 이전 대화 및 코드 상태를 복원하거나 선택한 메시지에서 요약하십시오.
- `**"Undo that"`\*\*: Claude에게 변경 사항을 되돌리도록 하십시오.
- `**/clear**`: 관련 없는 작업 간에 context를 재설정하십시오. 관련 없는 context가 있는 긴 세션은 성능을 줄일 수 있습니다.
- 한 세션에서 같은 문제에 대해 Claude를 두 번 이상 수정했다면 context는 실패한 접근 방식으로 복잡해져 있습니다. `/clear`를 실행하고 배운 내용을 통합하는 더 구체적인 프롬프트로 새로 시작하십시오. 누적된 수정이 있는 긴 세션보다 더 나은 프롬프트가 있는 깨끗한 세션이 거의 항상 더 나은 성능을 발휘합니다.
